Design Principles

P1
Mobile First

Small screen optimized layouts, large tap targets, simple navigation, bottom nav where useful, minimal scrolling, fast loading, offline-friendly, one-hand usage.

P2
Few Clicks to Learning

CBC: Choose Grade → Subject → Learn/Practice/Exam → Start. Tech: Choose Academy → Track → Topic → Learn/Practice/Visualize/Code. Users reach content in 3–4 taps.

P3
Framework First

Not only a CBC app — a reusable learning framework. Academies are added through configuration, module registration, content files, and optional overrides.

P4
Configuration Before Custom Code

Prefer configuration over hardcoded logic: theme tokens, navigation labels, academy structure, module structure, reward rules, exam scoring rules, unlock rules.

P5
Core Stability

The core never changes when a new academy, module, topic, or grade is added. New experiences are added through academy config, module config, content registry, theme config, and optional overrides.

Component Override System

Overrides follow a strict resolution order: module-specific first, then academy-specific, then the core default. Core components must be generic — no academy names, no grade names hardcoded.

academies/cbc/modules/grade-1/overrides/LessonCard.tsx
↓ fallback
academies/cbc/overrides/LessonCard.tsx
↓ fallback
core/components/LessonCard.tsx
❌ Bad: if (grade === "grade-1") showMascot()
✓ Good: if (theme.components.showMascot) showMascot()

Security Architecture

🔐
Authentication

Email/password, Google, Apple, Clever for schools. MFA required for admin accounts, optional for teachers.

🔒
Encryption

TLS 1.3 in transit · AES-256 at rest. No hardcoded secrets or credentials in source.

👤
RBAC

Academy-scoped permissions. Role: school-admin, teacher, IT-admin with granular control.

📋
Audit Logging

All authentication events, role changes, data exports, and consent actions are logged.

🛡️
Hardening

Annual penetration testing. Weekly dependency scanning. Rate limiting on all API endpoints.

📊
Compliance

COPPA, GDPR, FERPA considered. Consent collection, data export, and deletion on request.

Non-Negotiable Engineering Rules

01 No academy-specific logic inside core.
02 No hardcoded grade names inside reusable components.
03 No direct localStorage usage outside progress services.
04 No direct speech API usage inside UI components.
05 No direct payment logic inside UI components.
06 No duplicated screen logic across academies.
07 All content must pass schema validation.
08 All themes must use tokens — never inline or hardcoded values.
09 Overrides must follow the documented resolution order.
10 Essential learning content must never be locked behind gamification.
